Part 1: まずはここから！GitHubの基本

GitHubって何？ Gitとの違いは？

少しややこしいですが、GitとGitHubは別物です。

  • Git: あなたのパソコン上で動く「バージョン管理の魔法」です。ファイルの変更履歴を「セーブポイント」のように記録してくれます。間違えても過去に戻したり、変更点を見比べたりできる便利な道具です。
  • GitHub: Gitの魔法を、インターネット上で使えるようにしたWebサービスです。Gitで記録したセーブポイント付きのファイルを、安全に保管・共有できます。

Part 1: まずはここから！GitHubの基本

なぜGitHubを使うの？

  • 最強のバックアップ: あなたのコードがGitHub上にあるので、もしパソコンが壊れても安心です。
  • チーム作業がスムーズに: 誰がどこを直したか一目瞭然。複数人での開発がとても楽になります。
  • 自分の作品集になる: 作ったものを公開すれば、あなたのスキルを証明するポートフォリオになります。
  • 世界とつながる: 世界中の開発者が公開しているコードを読んだり、プロジェクトに参加したりできます。

Part 1: まずはここから！GitHubの基本

アカウントを作ってみよう

まずはGitHubの世界への扉を開きましょう。

  1. GitHub公式サイトにアクセスします。
  2. 画面右上の「Sign up」ボタンをクリック。
  3. 画面の指示に従って、メールアドレス、パスワード、ユーザー名を入力します。
  4. 届いた確認メールのリンクをクリックして、認証を完了させればOKです！

プランは「Free（無料）」で全く問題ありません。

Part 2: これだけは覚えたい！基本用語

専門用語が出てきますが、身近なものに例えると簡単です。

用語例えるなら…説明
Repository (リポジトリ)プロジェクトの「保管庫」ソースコードや画像など、プロジェクトの全ファイルを保管する場所。
Commit (コミット)「セーブポイントを作る」ファイルの変更を、メッセージ付きでリポジトリに記録する操作。
Push (プッシュ)「オンラインにアップロード」PC上のセーブポイントを、GitHub上の保管庫に送信すること。
Pull (プル)「最新版をダウンロード」GitHub上の最新の状態を、自分のPCに持ってくること。
Branch (ブランチ)「安全な作業用のコピー」メインのコードに影響を与えずに、新しい機能追加などを試すための「枝分かれ」機能。
Merge (マージ)「コピーした作業を合体」枝分かれさせたブランチでの作業が完了したら、それをメインのコードに統合すること。
Pull Request (プルリクエスト)「マージお願いします！という依頼」「この変更をマージしてください」と、他のメンバーにレビューをお願いする機能。
Fork (フォーク)プロジェクトを「自分用にコピー」他の人の公開リポジトリを自分のアカウントにコピーし、自由に編集できるようにする機能。

Part 3: 実際に使ってみよう！開発の流れを体験

Step 1: 準備編（自分のPCの設定）

  1. Gitをインストール: Git公式サイトからダウンロードし、インストールします。
  2. 初期設定: ターミナル等を開き、ユーザー名とメールアドレスを登録します。 
    git config --global user.name "あなたのGitHubユーザー名"
git config --global user.email "あなたのGitHubメールアドレス"

Part 3: 実際に使ってみよう！

Step 2: プロジェクトの箱を作る（リポジトリ作成）

  1. GitHubにログインし、右上の「+」から「New repository」を選択。
  2. Repository name に好きな名前（例: `hello-world`）を入力。
  3. Public（誰でも見れる）か **Private**（自分と招待した人だけ）かを選びます。
  4. 「**Add a README file**」にチェックを入れるのがおすすめです。
  5. 「Create repository」ボタンをクリックして完成！

Part 3: 実際に使ってみよう！

Step 3: コードをPCに持ってくる（クローン）

GitHub上に作った保管庫を、自分のPCに丸ごとコピーします。

  1. 作成したリポジトリのページで、緑色の「<> Code」ボタンをクリック。
  2. 表示されたURLをコピーします。
  3. ターミナルで、作業したいフォルダに移動し、以下のコマンドを実行します。 
    git clone コピーしたURL

Part 3: 実際に使ってみよう！

Step 4: コードを書いて記録する（編集 → add → commit）

  1. ファイル編集: クローンしてできたフォルダ内で、新しいファイルを作ったり編集します。
  2. 変更を記録対象に: どのファイルをセーブポイントに含めるかを選びます。 
    # 変更した全てのファイルを対象にする場合
git add .
  3. セーブポイント作成: メッセージを付けて変更を記録します。 
    git commit -m "初めてのコミット！"

Part 3: 実際に使ってみよう！

Step 5: 記録をGitHubに送る（プッシュ）

PC上で作ったセーブポイントを、GitHubにアップロードします。

git push origin main

これで、あなたの変更がGitHub上のリポジトリに反映されました！ブラウザで確認してみましょう。

Part 4: チーム開発の基本！

安全に作業するための「ブランチ」

完成しているメインのコード（`main`ブランチ）を直接いじるのは危険です。そこで、新しい作業を始めるときは「ブランチ」を切って、安全な場所で作業するのが基本です。

  1. ブランチ作成と移動: 
    git checkout -b feature/add-new-text
  2. 作業: このブランチで、自由にファイル編集、add、commitを行います。
  3. プッシュ: 作業が終わったら、この新しいブランチをGitHubにプッシュします。 
    git push -u origin feature/add-new-text

Part 4: チーム開発の基本！

変更を提案する「プルリクエスト」

ブランチでの作業が完了したら、「この変更をメインのコードに合体してください！」とお願いするのがプルリクエストです。

  1. ブランチをプッシュすると、GitHubのリポジトリページに「Compare & pull request」ボタンが表示されるのでクリック。
  2. 変更内容がわかるようにタイトルと説明を書き、「Create pull request」をクリック。
  3. チームメンバーが内容を確認（レビュー）し、問題なければ「Merge pull request」ボタンで、あなたの変更がメインのコードに無事統合されます。

セキュリティ：機密情報を守る (1)

"うっかり"を防ぐ！ .gitignore の活用

APIキーやパスワードなどの機密情報をGitHubにアップロードするのは非常に危険です。.gitignoreファイルは、特定のファイルをGitの追跡対象から除外するための設定ファイルです。

例:

# 設定ファイル
.env

# OSが自動生成するファイル
.DS_Store

# Node.jsの依存パッケージ
node_modules/

# ログファイル
*.log

重要: .gitignoreは、プロジェクトの最初期に設定し、機密情報が一度もコミットされないようにすることが肝心です。

セキュリティ：機密情報を守る (2)

漏洩検知ツールで二重にチェック

.gitignoreを設定していても、ミスは起こりえます。そこで、コミットやリポジトリの履歴に機密情報が含まれていないか、自動でスキャンするツールを導入するのがおすすめです。

  • gitleaks: リポジトリ内の全てのコミット履歴をスキャンし、正規表現に基づいてAPIキーやパスワードなどの機密情報を検出してくれるツールです。

    CI/CDパイプラインに組み込むことで、プッシュされる前に自動でチェックできます。

  • GitHub Advanced Security: GitHubが提供する高度なセキュリティ機能（有料プラン）です。Secret scanning機能が、リポジトリにプッシュされた機密情報を自動で検知し、警告してくれます。

発展：次のステップへ

GitHubの世界はもっと奥が深いです！基本をマスターしたあなたが次に挑戦したいのはこちら。

  • GitHub Flow / Git Flow: より体系的で大規模な開発にも対応できるブランチの運用ルールを学んでみましょう。
  • Issue (イシュー): タスク管理、バグ報告、機能の要望などをGitHub上で管理する機能です。開発の「やることリスト」として活用できます。
  • GitHub Actions: テストやデプロイといった作業を自動化する機能です。手作業を減らし、開発を効率化できます。
  • GitHub Pages: リポジトリから驚くほど簡単にWebサイトを公開できます。ポートフォリオサイトの作成に最適です。
  • OSSへの貢献: 世界中のオープンソースプロジェクトに参加してみましょう。小さな修正からでも、大きな学びがあります。
